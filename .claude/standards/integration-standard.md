## Synchronous Integration (REST)

### When to Use
- The caller requires an immediate response to continue processing
- Reading data from another context (query, not command)
- External-facing APIs consumed by clients or third parties

### Rules
- Use REST with OpenAPI 3 contracts ‚Äî no undocumented endpoints
- Use HTTP semantics correctly: GET (read), POST (create), PUT (replace), PATCH (partial update), DELETE (remove)
- Return RFC 7807 `ProblemDetail` for all error responses
- Version APIs: `/v1/`, `/v2/` ‚Äî never break an existing contract without a new version
- Set timeouts on all outbound HTTP calls (see timeout table below)
- Apply circuit breakers (Resilience4j) on all synchronous service calls
- Never expose internal service ports ‚Äî route through API Gateway or ALB

### Timeout Policy

| Call Type | Connect Timeout | Read Timeout | Notes |
|---|---|---|---|
| Internal service (same VPC) | 2s | 10s | Low latency; fast-fail if unavailable |
| Internal service (heavy computation) | 2s | 30s | Increase only with documented justification |
| External / third-party API | 5s | 30s | Never higher without explicit SLA requirement |
| Database query (OLTP) | 2s | 5s | Long queries are a design smell |
| Database query (reporting) | 2s | 120s | Reporting queries only; never on request path |

Configure via `application.yml` ‚Äî never hardcode in Java:

```yaml
app:
  http-client:
    connect-timeout: 2s
    read-timeout: 10s
```

### Retry Policy

Only retry on transient failures. Retrying client errors (4xx) is always wrong.

| HTTP Status | Retry? | Max Attempts | Strategy |
|---|---|---|---|
| 429 Too Many Requests | Yes | 3 | Respect `Retry-After` header; exponential backoff |
| 500 Internal Server Error | Yes | 3 | Exponential backoff with jitter |
| 502 Bad Gateway | Yes | 3 | Exponential backoff with jitter |
| 503 Service Unavailable | Yes | 3 | Exponential backoff with jitter |
| 504 Gateway Timeout | Yes | 2 | Longer delay; may indicate overload |
| 4xx (other) | No | ‚Äî | Client error; fix the request |

Backoff formula: `wait = min(base √ó 2^attempt + random(0, base), max_wait)`
- `base` = 1s, `max_wait` = 30s
- Add jitter to prevent thundering herd

```java
// Resilience4j retry configuration
@Bean
public RetryConfig defaultRetryConfig() {
    return RetryConfig.custom()
        .maxAttempts(3)
        .waitDuration(Duration.ofSeconds(1))
        .enableExponentialBackoff()
        .exponentialBackoffMultiplier(2)
        .retryOnException(e -> e instanceof ServiceUnavailableException)
        .ignoreExceptions(ClientErrorException.class)
        .build();
}
```

### Circuit Breaker (Resilience4j)

```yaml
resilience4j:
  circuitbreaker:
    instances:
      default:
        failure-rate-threshold: 50
        slow-call-rate-threshold: 80
        slow-call-duration-threshold: 2000ms
        wait-duration-in-open-state: 30s
        permitted-number-of-calls-in-half-open-state: 5
        sliding-window-size: 20
        minimum-number-of-calls: 10
        register-health-indicator: true
```

### Error Handling
```
2xx ‚Üí success
400 ‚Üí client error (do not retry)
401/403 ‚Üí auth error (do not retry)
404 ‚Üí not found (do not retry)
429 ‚Üí rate limited (retry with Retry-After backoff)
5xx ‚Üí server error (retry with exponential backoff + jitter, max 3 attempts)
```

### Idempotency Keys

All `POST` and `PATCH` operations that could be retried by clients must support idempotency keys.

- Clients send: `Idempotency-Key: <uuid>` request header
- Server stores the key + response for 24 hours
- Duplicate requests with the same key return the stored response without re-executing
- Key format: UUID v4 generated by the caller

```java
// Spring filter example ‚Äî check idempotency key before processing
@Component
public class IdempotencyFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response,
                                    FilterChain chain) throws IOException, ServletException {
        String key = request.getHeader("Idempotency-Key");
        if (key != null && isWriteMethod(request.getMethod())) {
            Optional<CachedResponse> cached = idempotencyStore.get(key);
            if (cached.isPresent()) {
                writeCachedResponse(response, cached.get());
                return;
            }
        }
        chain.doFilter(request, response);
    }
}
```

## Asynchronous Integration (Events)

### When to Use
- Cross-bounded-context communication where the producer does not need the result
- Long-running operations (decoupled processing)
- Fan-out scenarios (one event, multiple consumers)

### Rules
- Events are immutable facts: use past tense (`OrderPlaced`, `PaymentAuthorised`)
- Event schema must be versioned and registered in the Event Catalog (see `event-driven.md`)
- Events must include: `eventId` (UUID), `eventType`, `schemaVersion`, `timestamp` (ISO-8601 UTC), `correlationId`, `producerService`, `payload`
- Never put PII in event headers ‚Äî only in encrypted payload if required
- Consumers must be idempotent ‚Äî the same event may be delivered more than once
- Use the Outbox Pattern for transactional event publishing (never publish in a catch block)
- Every queue must have a Dead Letter Queue (DLQ); DLQ depth > 0 triggers an alarm

> For full event-driven patterns including schema evolution, DLQ setup, retry policy, and circuit breaker configuration, see `.claude/standards/event-driven.md`.

### AWS Messaging Selection
| Pattern | Use | AWS Service |
|---------|-----|-------------|
| Point-to-point queue | One consumer, guaranteed delivery | SQS |
| Fan-out | Multiple consumers | SNS ‚Üí SQS |
| Event bus (decoupled routing) | Dynamic routing rules | EventBridge |
| High-throughput streaming | Analytics, audit, replay | Kinesis |

## Anti-Patterns

- ‚ùå Shared databases between bounded contexts
- ‚ùå Synchronous chains longer than 3 hops (A‚ÜíB‚ÜíC‚ÜíD) ‚Äî use events
- ‚ùå Polling loops instead of event subscriptions
- ‚ùå Direct repository calls across service boundaries
- ‚ùå Undocumented REST endpoints
- ‚ùå Events without a registered schema
